# Umbra — Project Documentation

GitHub is the source of truth for this project documentation. Notion indexes this file in the Priyansh App Factory Command Center.

## 00. Executive Summary
Umbra is a native iOS visual camera app focused on creating a fast, shareable visual moment. It is for iPhone users who like playful visual tools and short creative experiences. The end product should include onboarding, one signature visual interaction, capture/save/share, local gallery, settings, and a landing page.

## 01. Product
MVP scope: onboarding, live preview, one signature effect, capture/save/share, local gallery, settings. Acceptance criteria: the app launches, the first-use flow is clear, and a user can create one output in under 60 seconds.

## 02. Design
Dark cinematic palette, glass-like controls, high contrast action button, minimal text, and clear first-run onboarding. Screens: onboarding, live preview, capture review, gallery, settings.

## 03. Frontend Technical
SwiftUI app shell with native iOS visual preview layer. Navigation: Onboarding, Preview, Capture Review, Gallery, Settings. Store saved items locally.

## 04. Backend Technical
No backend for v1. Future services may include effect catalog, challenge prompts, or share templates.

## 05. Business
Business model: free core experience with optional effect packs or pro unlock. Keep early operating cost low.

## 06. Marketing
Positioning: a fast visual playground for shareable moments. Channels: short demo videos, iOS communities, and creative app testers.

## 07. User Acquisition
Beta with 25-50 visual creators and iPhone users. Metrics: first-use completion, output creation, save/share rate, and D1 return.

## 08. Execution
Plan: audit repo, freeze MVP, build preview flow, add capture/share, QA, TestFlight.

## 09. QA
Test onboarding, preview, capture, save/share, gallery, app relaunch, device sizes, and accessibility labels.

## 10. Legal / Compliance
Document actual data handling and permission usage after implementation. Keep the first release simple and transparent.

## 11. Operations
Release process: internal build, small beta, TestFlight, launch decision. Post-launch: more effects, prompts, and share templates.
